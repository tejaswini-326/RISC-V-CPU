PARAMETERS IN VERILOG
<module> #(parameter <PARAMETER>)();
then while instantiating:
<module name> #(<param val>) <instance name>
MAKING THE NUMBER OF INPUTS DYNAMIC:
NO we canT do that >:(in 2001 we make a flattened multidimentional array and access via indexing)
in newer verilog(SYSTEM VERILOG) we can make a multidimensional array without flattening
module <module_name> #(parameter width = <default>, parameter num_inputs = <default>)(
    input [width-1:0] in [num_inputs-1:0]
);
if the inputs are from different sources, do this when instantiating:
    wire [31:0] a, b, c;
    wire [1:0] sel;
    wire [31:0] out;

    param_mux #(.NUM_INPUTS(3), .WIDTH(32)) my_mux (
        .in({c, b, a}),  // <- pack in reverse order cause verilog unpacks msb first i,e c will be set to mux input 2 here
        .sel(sel),
        .out(out)
    )
USING SYSTEM VERILOG FOR MUX.
SLICING IN SYSTEM VERILOG
somereg[start_point +: width] --> returns a slice of [start+width-1: start]
anotherreg[start_point -: width] --> returns a slice of [start: start- (width - 1)]

fixed issues with pipeline registers, found new issues from waveform

Key recurring issues:

Decode stage is sometimes reading old instruction bits → IF/ID register not flushed/stalled correctly.
Forwarding logic is incomplete: rs1/rs2 often get stale values.
Load-use hazard detection missing → no bubble insertion when needed.
Branch handling stalls unnecessarily or doesn’t flush at correct time.
HALT detection is delayed by 1 cycle → probably only checked in MEM/WB instead of EX stage.
